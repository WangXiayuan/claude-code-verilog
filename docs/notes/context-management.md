# 上下文过长：问题与解决方案

## 问题

Claude Code 会话越长，累积的对话历史越多，可用上下文窗口被逐渐撑满。结果是：

- 输出质量下降，遗漏细节
- 对早期讨论过的约束"记不住"
- 生成速度变慢

## 解决方案

### 1. 主动 `/compact` 压缩上下文

当感觉 Claude Code 反应变慢或遗漏信息时：

```
/compact
```

它会自动把对话历史压缩成摘要释放空间，效果立竿见影。

### 2. 周期性 `/clear` 开启新会话

一个复杂功能拆成多个独立会话，每次只聚焦一小块：

```
会话1：搭好模块框架和接口定义
会话2：实现子模块A
会话3：实现子模块B
会话4：写 testbench + 联调
```

**关键**：每次新会话开始，用 `@` 引用上一阶段的产出文件，Claude 就能跟上进度。

### 3. 把重复要求写进 CLAUDE.md

对于每次都要强调的规则（编码风格、命名规范、仿真工具等），写入项目根目录 `CLAUDE.md`。Claude Code 每次启动自动读取，无需每次重复。

你省下的上下文 = 更高质量的输出。

### 4. 善用 Memory 做跨会话记忆

```
记住：本项目的目标器件是 Kintex-7，时钟 200MHz
记住：所有 RTL 放在 rtl/ 目录，testbench 放在 tb/ 目录
```

这些信息在 `/clear` 后依然保留。

### 5. 用 Agent 隔离子任务

复杂任务可以派生子 agent 独立执行各自模块，互不污染上下文：

```
# 同时写两个独立模块
Agent1 → 写 UART TX
Agent2 → 写 UART RX
```

### 6. 判断标准：何时该刷新

出现以下信号就该 `/clear` 或 `/compact` 了：

- 你问 A 它回答 B（注意力漂移）
- 生成速度明显变慢
- 同一个约束你重复了三遍以上
