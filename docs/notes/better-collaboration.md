# 如何与 Claude Code 配合越来越默契

## 核心思路

Claude Code 不是一次性工具，而是一个会随使用深入越来越懂你的搭档。关键在于：**持续给它积累关于你的上下文**。

## 1. 写好自己的 CLAUDE.md

项目根目录的 `CLAUDE.md` 是给 Claude Code 的"项目说明书"。每次启动会话它都会读这个文件。

```markdown
## 协作准则
收到任何需求时，先头脑风暴，提问以理解需求再动手。

## 编码偏好
- 组合逻辑用 always_comb，时序逻辑用 always_ff
- 每个 always 块只驱动一个信号
- 不要用 initial 块（testbench 除外）
```

**原则**：把你反复强调的要求写进 CLAUDE.md，以后就不用每次都重复了。

## 2. 善用 Memory 功能

对话中直接告诉 Claude Code 记住关键信息：

```
记住：我习惯用 Icarus Verilog 做仿真，目标器件是 Xilinx Kintex-7
记住：我的模块命名风格是 lowercase_snake_case
```

Claude Code 会把记忆保存下来，下次对话自动加载。

## 3. 给出具体的反馈

```
# ❌ 模糊
不对，重新写

# ✅ 具体
这个 always_comb 里的 case 缺少 default 分支，会产生 latch，补上
```

反馈越具体，Claude Code 调整方向越精准。你纠正一次，它在同一次会话中就会记住你的风格偏好。

## 4. 分阶段推进复杂任务

```
# 第一阶段：列方案
"我要做一个 UART 收发器，先别写代码，帮我列出模块拆分方案"

# 第二阶段：逐个实现
"方案没问题，先实现波特率发生器模块"

# 第三阶段：联调
"把写好的 baud_gen.v、uart_tx.v、uart_rx.v 集成到一个 top 模块"
```

不要让 Claude Code 一口吃成胖子。分步走质量更高，而且你对每个环节的掌控感更强。

## 5. 引用项目文件做上下文

把相关文件用 `@` 引进来，Claude Code 会自动理解代码风格和接口规范：

```
参考 @rtl/spi_master.v 的编码风格，写一个 I2C master 模块
```

比文字描述更高效——代码本身就是最精确的风格样本。

## 6. 养成总结习惯

每次会话结束前，花 30 秒让它总结：

```
总结一下今天这个模块设计中的关键决策和踩坑点
```

把有价值的总结写成笔记，下次遇到类似需求直接把笔记 @ 进来做参考。

## 7. 持续更新 CLAUDE.md

每当你发现"这句话我已经说了好几遍了"的时候，把它加进 CLAUDE.md。这样你的 CLAUDE.md 会越来越贴合你的工作习惯，Claude Code 也自然越来越顺手。
